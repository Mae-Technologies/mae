# Database Table Factory, ACL, and Audit System (PostgreSQL)

This document explains how to use the PostgreSQL table factory, audit enforcement, and access-control system defined by the following migrations:

* `00_create_status_enum.sql`
* `01_create_function_audit_enforce_timestamps_and_immutables.sql`
* `02_create_table_function.sql`
* `03_create_function_roles_priviledges.sql`
* `04_create_function_table_acl.sql`
* `05_lockdown_table_acl.sql`

The system is designed to:

* Safely create tables from structured specifications (no raw SQL injection)
* Enforce immutable audit fields
* Apply least-privilege access automatically
* Separate migration, provisioning, and runtime responsibilities cleanly

---

## High-level architecture

1. **Tables are created via a controlled factory function** (`create_table_from_spec`)
2. **Audit behavior is enforced at the database level** via triggers
3. **Column-level privileges are applied automatically**
4. **Only a restricted role can create tables**
5. **Application code runs with minimal privileges**

---

## Roles

### `app_owner` (NOLOGIN)

* Owns all generated tables
* Owns all `SECURITY DEFINER` functions
* Never used directly by the application

### `app_user` (NOLOGIN)

* Base role for runtime application access
* Granted:

  * `SELECT` on tables
  * Limited `INSERT` / `UPDATE` on allowed columns only
* All application login roles must inherit this role

### `table_creator` (NOLOGIN)

* Can execute:

  * `create_table_from_spec(jsonb)`
  * `apply_table_acl(...)`
* Cannot run arbitrary DDL
* Intended for migrations or controlled provisioning flows

### Recommended login roles

These are **not** created by the migrations and should be created separately:

| Role                           | Purpose                            |
| ------------------------------ | ---------------------------------- |
| `db_migrator`                  | Runs SQLx migrations               |
| `app_runtime`                  | Application database user          |
| `table_provisioner` (optional) | Runtime table creation (if needed) |

Grants:

```sql
GRANT app_user TO app_runtime;
GRANT table_creator TO table_provisioner;
```

## Default columns on every generated table

All tables created by `create_table_from_spec` include the following columns automatically.

### Identity / primary key

* `id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`

### Tenant / domain fields

* `sys_client INT NOT NULL`

  * Set on INSERT
  * Immutable after insert

* `status public.status NOT NULL`

  * User-writable

* `comment TEXT`

  * User-writable

* `tags JSONB NOT NULL DEFAULT '{}'::jsonb`

  * User-writable

* `sys_detail JSONB NOT NULL DEFAULT '{}'::jsonb`

  * User-writable

### Audit fields

* `created_by INT NOT NULL`

  * Insertable by application
  * Immutable after insert

* `updated_by INT NOT NULL`

  * Insertable and updatable by application

* `created_at TIMESTAMPTZ NOT NULL DEFAULT now()`

  * Immutable after insert

* `updated_at TIMESTAMPTZ NOT NULL DEFAULT now()`

  * Automatically updated on INSERT and UPDATE

---

## Audit and immutability enforcement

The trigger function `audit_enforce_timestamps_and_immutables()` enforces:

* On INSERT:

  * `updated_at` is set automatically
* On UPDATE:

  * `updated_at` is refreshed
  * Changes to the following columns are rejected:

    * `id`
    * `sys_client`
    * `created_by`
    * `created_at`

This enforcement applies regardless of client or privilege level.

## Creating tables

### Table factory function

```sql
public.create_table_from_spec(jsonb)
```

This function:

* Validates identifiers
* Resolves types using `to_regtype`
* Creates the table
* Attaches the audit trigger
* Automatically applies default ACLs

### JSON specification format

```json
{
  "table_name": "tasks",
  "columns": [
    { "name": "title", "type": "text", "nullable": false },
    { "name": "priority", "type": "int4", "default": 0 },
    { "name": "due_at", "type": "timestamptz" }
  ]
}
```

Supported column attributes:

* `name` (required)
* `type` (required, resolved via `to_regtype`)
* `nullable` (optional, default `true`)
* `default` (optional; JSON scalar only)
* `unique` (optional)

Not supported by design:

* Arbitrary SQL expressions
* Foreign keys
* Index definitions
* Computed/generated columns

## Automatic ACL application

After table creation, `create_table_from_spec` automatically calls:

```sql
apply_table_acl(table_name, insertable_extras, updatable_extras)
```

### Default permissions applied

**SELECT**

* Granted on the entire table (all columns readable)

**INSERT allowed**

* `sys_client`
* `status`
* `comment`
* `tags`
* `sys_detail`
* `created_by`
* `updated_by`
* All columns defined in the spec

**UPDATE allowed**

* `status`
* `comment`
* `tags`
* `sys_detail`
* `updated_by`
* All columns defined in the spec

**UPDATE disallowed**

* `id`
* `sys_client`
* `created_by`
* `created_at`
* `updated_at`

**Sequence access**

* `USAGE, SELECT` granted on the identity sequence

---

## Hardening permissions later

If a developer wants to **restrict additional columns** (e.g. make a spec column insert-only), they can re-run `apply_table_acl` in a later migration:

```sql
SELECT public.apply_table_acl(
  'tasks',
  ARRAY['title', 'priority'],  -- insertable extras
  ARRAY['title']               -- updatable extras
);
```

This overwrites previous column-level grants.

## SQLx configuration

### Migrations

Use a privileged connection string:

```bash
DATABASE_URL=postgres://db_migrator:password@host:5432/db
sqlx migrate run
```

The migrator must be able to:

* Create types, functions, roles
* Change function ownership
* Revoke/grant privileges

### Runtime application

Use a restricted user:

```bash
DATABASE_URL=postgres://app_runtime:password@host:5432/db
```

This role:

* Inherits `app_user`
* Cannot create tables
* Cannot modify protected columns

### Optional: runtime table provisioning

Only if required:

```bash
TABLE_CREATOR_DATABASE_URL=postgres://table_provisioner:password@host:5432/db
```

Used exclusively for calling `create_table_from_spec`.

## Troubleshooting

### `permission denied for sequence <table>_id_seq`

* `apply_table_acl` was not run
* Re-run `apply_table_acl` for the table

### `permission denied for function create_table_from_spec`

* Role lacks `table_creator`
* Verify `05_lockdown_table_acl.sql` ran

### `sys_client is immutable`

* Attempted UPDATE of `sys_client`
* Must only be set on INSERT

### `created_by is immutable`

* Attempted UPDATE of `created_by`
* Set only during INSERT

### `unknown or invalid type`

* `to_regtype` could not resolve the type
* Use canonical names or schema-qualify custom types

## Design notes

* All security-sensitive behavior is enforced **in the database**
* Application code never needs DDL privileges
* Table creation is auditable and repeatable
* Column-level grants provide fine-grained control
* The system is compatible with SQLx offline mode and migrations
